# Markdown to HTML Converter 実装ロードマップ（全22ステップ）

## Phase 1: 大枠づくり（5ステップ）
目的: まず「入力→変換→表示/DL」が確実に動く最小構成を成立させる

1. 入力画面の最小UIを用意する
- Markdown入力欄
- 出力モード選択（preview / download）
- Convertボタン

2. 変換エンドポイントを最小実装する
- POST受け取り
- Markdown文字列取得
- mode取得（previewをデフォルト）

3. Markdown変換処理を接続する
- Parsedown経由でMarkdown→HTML変換
- 最小の変換クラス経由で呼び出し

4. 出力2モードを実装する
- preview: HTMLを画面表示
- download: HTMLファイルとして返却

5. 実行確認（Phase 1完了判定）
- GET / が表示される
- POST preview が200で変換結果を返す
- POST download が200で添付ダウンロードになる


## Phase 2: 安全性・安定性の補強（8ステップ）
目的: 壊れにくく、安全な失敗をする実装へ段階的に補強する

1. レスポンス安全ヘッダーを追加
- nosniff
- frame拒否
- referrer policy
- 最小CSP

2. CSRFトークンをフォームに埋め込む
- セッションにトークン生成
- hidden inputで送信

3. CSRFトークンをサーバー側で検証
- 不一致/欠落時は安全に拒否
- エラー時の戻り動線を固定

4. 入力バリデーション強化
- 空文字拒否
- 長さ上限チェック
- modeの許可値チェック

5. エラーコード管理の明確化
- 許可されたエラーコードのみ扱う
- 未知のエラーコードは無視

6. XSS前提をコードで固定化
- 変換器のsafe mode前提を明文化
- 画面表示時のエスケープ責務を明確化

7. ダウンロード応答の安定化
- Content-Type
- Content-Disposition
- 文字コード方針

8. 異常系の確認観点を定義
- 空入力
- 上限超過
- 不正mode
- 不正トークン


## Phase 3: リファクタリング（5ステップ）
目的: 動作を変えずに、読みやすく保守しやすい構造へ改善する

1. 入力取得と前処理を整理
- 生データ取得
- 正規化
- デフォルト適用

2. バリデーション処理を関数化
- 条件分岐の重複を削減
- エラー返却の統一

3. エラー表示ロジックを一元化
- コード→文言マップ化
- 表示判定を単純化

4. 変換成功/失敗レスポンス生成を分離
- 正常系の見通し改善
- 失敗系の挙動を追いやすくする

5. 命名と最小コメントを整備
- 意図が伝わる関数名・変数名へ
- 複雑箇所に限定して短い説明コメント


## Phase 4: 追加改善（4ステップ）
目的: 品質を継続的に保てる運用基盤を最小コストで追加する

1. 手動テストチェックリストを作成
- 正常系/異常系/境界値を明記
- 変更後の回帰確認を標準化

2. 最小の自動テスト導入
- エンドポイント疎通テスト
- preview/downloadの基本検証

3. 設定値の外出し
- 最大文字数などを定数/設定へ分離
- 将来変更を安全化

4. デプロイ・運用メモの補足
- 起動手順
- 動作確認手順
- 典型的なトラブルシュート

## 全体の進め方（実務向け）
- 1ステップごとに「1変更テーマ」を守る
- 各ステップ終了時に実行確認
- 問題が出たら直前ステップに戻って修正
- Phase 2完了までは機能追加より安全性を優先
